# AI 驅動之字體風格解析與臨摹系統

這是一個整合 AI 視覺辨識、傳統影像渲染與物理特徵比對的 Web 應用系統。旨在讓使用者上傳任意字體圖片後，系統能解析其風格，動態生成相應風格的參考文字，並提供互動式的臨摹與評分環境。

---

## 1. 專案程式主要功能

* **AI 字體風格解析**：透過視覺語言模型 (VLM)，精準辨識使用者上傳圖片的字體風格、情緒與特徵。
* **動態風格化文字生成**：使用者輸入任意文字後，系統能根據解析出的風格，結合本地端開源字體庫與影像處理演算法，動態渲染出帶有特定風格的參考圖片。
* **互動式臨摹畫布**：提供 Web 端的互動畫布，支援滑鼠或平板筆，讓使用者直接在網頁上進行手寫練習。
* **雙重評分與回饋機制**：
    * **客觀幾何評分**：計算手寫筆跡與標準參考圖的物理重疊度 (如 SSIM)。
    * **AI 結構點評**：提供針對筆劃結構、佈局比例的自然語言改進建議(此樣可選，若時間緊迫可以不開發)。

---

## 2. 專案運作流程

1. **影像上傳**：
   * 使用者於前端上傳一張字體圖片。
2. **影像字體解析**
   * 後端將圖片發送至 AI API，取得該字體的概要描述及結構化特徵參數 (JSON 格式，包含基礎字體分類、粗細、邊緣狀態等)。
3. **使用者輸入欲練習文字**：
   * 使用者輸入想練習的文字（例如：「你好」）。
4. **參考練習字生成**
   * 系統依據 AI 解析的「基礎字體分類」，從本地庫提取對應的 `.ttf` 檔案，利用 Pillow 渲染出標準圖片。
   * 系統再依據 AI 解析的「特徵參數」，利用 OpenCV 加上對應濾鏡（例如：暈染、加粗、變形），產出最終風格化參考圖並回傳前端。
5. **實作臨摹**：
   * 使用者參考生成的圖片，在 Web 畫布上書寫，或上傳手寫成果圖。
6. **比對與回饋**：
   * 後端接收手寫圖，使用 OpenCV 進行影像對齊與相似度計算，得出客觀分數。
   * 將參考圖與手寫圖同步送交 AI，取得主觀的結構改進建議。
   * 前端統整並展示最終的綜合分數與點評。

---

## 3. 專案系統架構

本專案採用前後端分離，並結合雲端 AI 與本地渲染引擎的混合架構：

* **前端介面 (Web UI)**：
    * 採用 HTML/CSS/JavaScript 實作，包含上傳元件、動態表單與 HTML5 `<canvas>` 繪圖區。
* **後端核心 (Python / Flask)**：
    * 使用python實作，作為系統樞紐，處理 API 請求、業務邏輯，並負責各模組間的資料傳遞。
* **本地字體資料庫 (Font Library)**：
    * 存放 20~30 套精選開源中文字體 (`.ttf` / `.otf`)。
    * 維護 `registry.json` 屬性註冊表，供程式進行特徵匹配。
* **影像渲染引擎 (Pillow + OpenCV)**：
    * 處理底層的文字轉圖片，以及後續的像素級濾鏡後處理。
* **AI 視覺引擎 (Cloud API)**：
    * 串接具備視覺能力的大型語言模型 (如 Gemini 1.5 Pro 或 GPT-4o)，處理風格萃取與文字回饋。

## 4. 專案依賴的 Python 套件

建議在獨立的虛擬環境 (如 Miniconda 或 venv) 中安裝以下核心套件：

```text
# 後端 Web 框架 (此區域可以先不實裝，待之後有需求時再實裝)
Flask==3.0.2         # 輕量級 Web 伺服器
flask-cors==4.0.0    # 處理跨網域資源共用 (CORS) 請求

Pillow==10.2.0       # 負責讀取字型檔並渲染基礎文字圖片

numpy==1.26.4        # OpenCV 底層資料結構依賴

# AI 串接 SDK (本專案使用Gemini)
google-generativeai==0.4.1  # 串接 Google Gemini API

# 環境變數管理
python-dotenv==1.0.1 # 安全讀取 .env 檔中的 API Keys 等機密資訊
```

## 5. 專案開發時會需要做的事情

* **建置本地字體庫**：蒐集並下載涵蓋不同風格 (黑體、明體、手寫、書法等) 的開源字體檔案。
* **標註 `registry.json`**：手動為下載的每一套字體定義屬性標籤 (如：`style`, `weight`, `mood`)，建立匹配基準。
* **設計與開發 OpenCV 濾鏡模組**：撰寫獨立的 Python 函數來模擬字體特效 (例如撰寫 `apply_ink_bleed()` 來模擬墨水暈染、`apply_thickness()` 調整粗細)。
* **撰寫 System Prompt (提示詞工程)**：設計嚴謹的 Prompt，確保 AI 能穩定分析圖片並輸出符合後端邏輯格式的 JSON 資料。
* **前後端 API 串接實作**：建立 Flask 路由，並在前端實作 Canvas 的軌跡紀錄、影像 Base64 編碼轉換與非同步請求 (Fetch/Axios)。

---

## 6. 專案開發時會遇到的困難點

* **AI 輸出的不穩定性 (幻覺處理)**：
  AI 有時可能未嚴格遵守 Prompt 指示，輸出錯誤的 JSON 格式或不在清單內的字體標籤。
  *對策*：後端需實作強健的資料驗證與 Try-Catch 機制，若解析失敗需有自動退回使用「預設標準字體」的防呆設計。
* **OpenCV 參數調校的複雜度**：
  要用影像演算法逼真地模擬出「人類手寫的瑕疵感」或「毛筆暈染」並不容易，且同一套濾鏡參數套用在不同字體上效果可能差異極大。
  *對策*：需開發一個本地端的視覺化測試腳本，利用滑桿即時預覽不同濾鏡參數的組合效果，再將最佳參數寫入後端邏輯中。
* **影像對齊與評分精準度**：
  前端 Canvas 擷取的手寫圖，其尺寸、文字位置或比例可能與後端生成的參考圖不一致，直接進行像素比對 (如 SSIM) 會導致評分嚴重失準。
  *對策*：在進行相似度計算前，需實作影像前處理邏輯 (如 Bounding Box 擷取、影像縮放對齊、背景去背與二值化)，確保兩張圖片在相同的基準點上進行比較。
